# 07. 자동 PR·자동 머지 세팅 가이드

> 오븐업 **코드 레포(`Oven-up`)** 에 CI 검사 + 자동 머지를 붙이는 방법. 파일은 `docs/github-setup/` 에 준비돼 있다.
> 대상: 입문자 · 버전: v0.1 · 작성일: 2026-07-02

---

## 먼저 알아둘 것

- **제가(문서로) 준비한 것**: 워크플로 파일들(`github-setup/`).
- **직접 하셔야 하는 것**: 파일을 코드 레포에 올리고, GitHub 웹에서 몇 가지 설정을 **클릭**으로 켜는 일. (계정 설정은 대신 못 켭니다)
- 순서를 그대로 따라오시면 됩니다.

## 1단계. 코드 레포 시작 + 브랜치 만들기

컴퓨터에서 (경로는 예시):
```bash
git clone https://github.com/Tae0072/Oven-up.git
cd Oven-up

# 첫 커밋(빈 레포라 main 브랜치 생성용)
echo "# Oven-up" > README.md
git add README.md
git commit -m "chore: 레포 초기화"
git branch -M main
git push -u origin main

# 개발 통합용 dev 브랜치 생성
git checkout -b dev
git push -u origin dev
```

## 2단계. 세팅 파일 복사 + 올리기

`docs/github-setup/` 안의 파일을 **코드 레포(Oven-up) 루트**로 복사한다.
(탐색기에서 점(.)으로 시작하는 파일이 안 보이면 "숨긴 항목 표시"를 켜세요.)

복사 후 코드 레포 구조:
```
Oven-up/
├── .github/
│   ├── CODEOWNERS
│   ├── pull_request_template.md
│   └── workflows/
│       ├── ci.yml
│       ├── auto-merge.yml
│       └── claude-review.yml   (선택, 안 쓸 거면 빼도 됨)
├── .gitignore
└── README.md
```

dev 브랜치에 올리기:
```bash
git checkout dev
git add .
git commit -m "chore: CI·자동머지·PR템플릿 추가"
git push
```

## 3단계. GitHub 설정 켜기 (웹에서 클릭)

레포 페이지 → **Settings** 에서:

### (1) 자동 머지 허용
- **Settings → General → Pull Requests**
- ✅ **Allow auto-merge** 체크
- (권장) ✅ Allow squash merging / ✅ Automatically delete head branches

### (2) Actions 권한
- **Settings → Actions → General → Workflow permissions**
- ✅ **Read and write permissions** 선택 (자동 머지 워크플로가 PR을 다루기 위해 필요)

### (3) 기본 브랜치
- **Settings → General → Default branch** 를 **dev** 로 바꾸면 PR이 기본적으로 dev를 향한다. (선택)

## 4단계. 필수 검사(CI) 등록 → 브랜치 보호

자동 머지는 "필수 검사가 통과하면" 머지되는 구조라, **먼저 검사 이름을 한 번 실행해 등록**해야 한다.

### (1) 검사 이름 만들기 (첫 PR)
```bash
git checkout dev
git checkout -b feature/test-ci
echo "test" >> README.md
git add . && git commit -m "chore: CI 동작 테스트"
git push -u origin feature/test-ci
```
- GitHub에서 `feature/test-ci → dev` PR을 만든다.
- **Actions** 탭에서 `server-build`, `app-build`, `secret-scan` 이 실행되는지 확인.

### (2) dev 브랜치 보호 규칙
- **Settings → Branches → Add branch ruleset(또는 Add rule)**
- Branch name pattern: `dev`
- ✅ **Require a pull request before merging**
- ✅ **Require status checks to pass** → 검색해서 `server-build`, `app-build`, `secret-scan` 추가
- (선택) ✅ Require approvals: 혼자면 0, 팀이면 1
- 저장

> 이제부터 dev로 가는 PR은 세 검사가 모두 초록불이 되면 **자동으로 squash merge** 된다.

## 5단계. (선택) Claude 자동 리뷰

원하면 PR을 Claude가 자동 리뷰하게 할 수 있다.
1. GitHub Marketplace에서 공식 **Claude** GitHub 앱(Claude Code Action) 설치
2. **Settings → Secrets and variables → Actions → New repository secret**
   - 이름: `ANTHROPIC_API_KEY`, 값: 발급받은 API 키
3. `claude-review.yml` 을 최신 공식 문서 기준으로 옵션 조정
- 자동 리뷰가 필요 없으면 `claude-review.yml` 은 안 넣어도 된다. (CI + 자동 머지는 그대로 동작)

## 6단계. 동작 확인

```bash
git checkout dev && git pull
git checkout -b feature/menu-list
# ... 작업 ...
git add . && git commit -m "feat(menu): 메뉴 목록 화면 추가"
git push -u origin feature/menu-list
```
- PR 생성 → CI 실행 → 전부 통과 → **자동으로 dev에 머지 + 브랜치 삭제**.

## 자주 겪는 문제 (트러블슈팅)

| 증상 | 원인 / 해결 |
| --- | --- |
| 자동 머지가 안 됨 | 3-(1) Allow auto-merge 미체크 / 4-(2) 브랜치 보호에 검사 미지정 |
| auto-merge 워크플로 실패 | 설정 전이라 그럴 수 있음. 3~4단계 완료 후 재시도 |
| 검사 이름이 브랜치 보호에 안 보임 | 그 검사가 아직 한 번도 안 돌아서임. 4-(1)로 PR 한 번 실행 |
| PR 권한 오류 | 3-(2) Read and write permissions 확인 |
| gitleaks가 키를 발견 | 코드에 API 키/비밀번호가 들어감. 제거 후 키 재발급 |

## 요약
1. main·dev 브랜치 생성 → 2. `github-setup/` 파일 복사·push → 3. Allow auto-merge + Actions 권한 → 4. 첫 PR로 검사 등록 후 dev 브랜치 보호에 필수 검사 지정 → (5. 선택 Claude 리뷰) → 6. 확인.
