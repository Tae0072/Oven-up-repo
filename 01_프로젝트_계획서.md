# 01. 오븐업(Oven-up) 프로젝트 계획서

> 카페 주문·결제 앱/웹. 배달의민족처럼 메뉴를 보고, 장바구니에 담고, 주문·결제하고, 주문 내역을 조회하는 서비스. 예약·단체주문·고객의소리·관리자·쿠폰/적립까지 포함.
> 버전: v0.2 · 작성일: 2026-07-02 (개정: 소셜로그인·예약·단체주문·고객의소리·관리자·푸시·쿠폰/적립 범위 확대)

---

## 1. 한 줄 정의

**"메뉴 선택 → 장바구니 → 주문 → 결제 → 주문 조회"** 흐름을 웹과 앱에서 동일하게 제공하는 카페 주문 서비스.

여기에 **예약·단체주문·고객의소리** 같은 손님 편의 기능과, **사장님용 관리자 화면·쿠폰/적립·푸시 알림**까지 포함한다. 광고성 기능만 제외한다.

---

## 2. 만들 것 / 안 만들 것 (범위)

### ✅ 만들 것

**손님 핵심 흐름**
- 홈(홈페이지) — 메뉴 목록과 분리된 대문 화면
- 메뉴 목록(별도 페이지) / 메뉴 상세(옵션 선택)
- 장바구니 담기 / 수량 변경 / 삭제
- 주문하기 — 수령 방식(매장/포장/**배달**), 수령 시점(지금/**예약**), 요청사항
- 결제 — 카드 + **카카오·네이버·토스·삼성페이** 등 간편결제
- 주문 내역 조회
- 로그인 / 회원가입 + **카카오·네이버 소셜 로그인**

**손님 편의 기능**
- 예약 주문 (시간 지정 → 선결제 → 픽업/배달)
- 단체 주문
- 고객의 소리 (문의·건의 + 답변)
- 마이페이지 (주문·쿠폰·적립·예약 내역)

**혜택·알림**
- 쿠폰 / 적립
- 푸시 알림 (주문 상태, 예약·문의 답변 등)

**사장님용 관리자**
- 메뉴 관리 / 주문 접수·상태 변경 / 예약·단체·고객의소리 관리 / 쿠폰 관리

### ❌ 안 만들 것 (이번 범위 제외)
- 광고, 배너
- 리뷰, 별점
- 다지점(여러 매장) 지원

> 기능이 늘어난 만큼 **한 번에 다 만들지 않고**, 손님 핵심 흐름 → 편의 기능 → 관리자·결제·알림·혜택 순서로 단계별로 붙인다(4번 로드맵 참고).

## 3. 기술 스택 (모놀리식 · 입문자 기준)

참고 레포(2nd-Team-Project)는 마이크로서비스(MSA)라는 복잡한 방식이지만, **우리는 단순한 모놀리식(하나의 앱) 방식**으로 만든다. 입문자가 혼자 다루기 쉽고, 카페 주문 정도 규모엔 충분하다.

| 구분 | 선택 | 이유 |
| --- | --- | --- |
| 프론트엔드 (화면) | **Flutter** | 코드 하나로 **웹 + 안드로이드 + iOS** 모두 만들 수 있어 "웹과 앱 둘 다"에 딱 맞음. 팀 참고자료도 있음 |
| 백엔드 (서버) | **Spring Boot (단일 앱)** | 참고 레포와 언어가 같아 자료 활용 가능. MSA로 쪼개지 않고 **하나의 서버**로 운영 |
| 데이터베이스 | **MySQL** | 가장 널리 쓰이는 무료 DB. 자료가 많음 |
| 결제 | **PortOne(구 아임포트)** | 카드 + **카카오·네이버·토스·삼성페이** 등 간편결제를 한 곳으로 연동. 예약·배달 주문도 동일 결제 사용 |
| 소셜 로그인 | **카카오·네이버 OAuth** | 카카오/네이버 developers에 앱 등록 후 로그인 연동 |
| 푸시 알림 | **Firebase Cloud Messaging(FCM)** | 주문 상태·답변 알림 전송 |

> 💡 **더 쉬운 대안**: 서버(Spring Boot)와 DB를 직접 다루는 게 부담되면, **Firebase**(구글의 백엔드 대행 서비스)로 회원가입·데이터 저장·소셜 로그인·푸시를 한 번에 처리할 수도 있다. 서버 코드를 거의 안 짜도 된다. 학습 부담을 더 줄이고 싶으면 이 방향도 고려.

> ⚠️ **외부 등록이 필요한 것들**: 카카오·네이버 로그인, 결제(PortOne 등), 푸시(FCM)는 각 사이트에 가입·앱 등록·심사가 필요하다. 그래서 화면과 기본 흐름을 먼저 만들고 **나중에 연결**하는 걸 권장.

### 왜 이렇게 구성하나? (아주 쉽게)
- **프론트엔드** = 손님 눈에 보이는 화면 (메뉴, 버튼 등)
- **백엔드(서버)** = 화면 뒤에서 주문을 받고 저장하고 처리하는 부분
- **데이터베이스** = 메뉴, 주문, 회원 정보를 실제로 저장하는 창고
- **결제 연동** = 카드 결제는 직접 못 만들고, 전문 업체(PortOne 등)에 맡긴다

## 4. 전체 개발 순서 (로드맵)

입문자 기준으로 **한 번에 하나씩** 완성하며 나아간다.

### 0단계. 준비 (지금 하는 중)
- [x] 기획·설계 문서 작성 ← **현재 여기**
- [ ] 개발 환경 설치 (Git, 에디터, Flutter, JDK 등)
- [ ] 두 레포(Oven-up 코드, Oven-up-repo 문서) 컴퓨터에 내려받기
- [ ] PR 규칙(.github) 세팅

### 1단계. 손님 화면만 먼저 (데이터 없이)
- [ ] 홈(홈페이지) 화면
- [ ] 메뉴 목록(별도 페이지) — 가짜 데이터
- [ ] 메뉴 상세 / 옵션 선택
- [ ] 장바구니 / 주문서 (UI만)

### 2단계. 서버 + 데이터베이스
- [ ] DB 설계대로 테이블 만들기 (회원·메뉴·주문 등)
- [ ] 메뉴 목록 API
- [ ] 주문 저장 API
- [ ] 로그인/회원가입 API (자체 로그인 먼저)

### 3단계. 화면 + 서버 연결
- [ ] 화면이 서버에서 진짜 메뉴를 가져오도록
- [ ] 장바구니 → 주문 → 서버 저장 흐름 완성
- [ ] 주문 내역 조회 · 마이페이지 연결

### 4단계. 손님 편의 기능
- [ ] 예약 주문 (시간 지정 + 배달 주소) — 결제 연동 후 완성
- [ ] 단체 주문
- [ ] 고객의 소리

### 5단계. 관리자(사장님) 화면
- [ ] 관리자 로그인(권한 분리) + 대시보드
- [ ] 메뉴 관리 (등록/수정/품절)
- [ ] 주문 관리 (접수/상태 변경)
- [ ] 예약·단체·고객의소리 관리

### 6단계. 결제 · 소셜 로그인 · 알림 · 혜택
- [ ] 결제 연동 (PortOne 등) + 서버 검증
- [ ] 카카오·네이버 소셜 로그인
- [ ] 푸시 알림 (FCM)
- [ ] 쿠폰 / 적립

### 7단계. 마무리
- [ ] 버그 수정, 테스트
- [ ] 배포 (웹 / 앱 스토어)

> 각 단계 안에서도 "가장 작은 것부터" 만든다. 예를 들어 메뉴 화면 하나가 완전히 돌아가는 걸 먼저 확인하고 다음으로 넘어간다.

## 5. 작업 규칙 (지침)

- **작업 전 항상 최신화**: 코드를 짜기 전에 `git pull`로 최신 상태를 먼저 받는다.
- **코드는** `Oven-up` 레포에, **문서는** `Oven-up-repo` 레포에 저장한다. (섞지 않기)
- **PR 규칙**은 `QT-AI-2nd-Team-Project`의 `.github` 규칙을 그대로 적용한다.
- 문서 작성 시 `2nd-Team-Project`는 **구조·형식만 참고**하고, 카페 앱에 필요한 것만 가져온다.
