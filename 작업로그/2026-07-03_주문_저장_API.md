## 2. 리포트 (PR #13)

### API (05_API §4)
| 메서드/경로 | 설명 | 성공 | 실패 |
| --- | --- | --- | --- |
| POST /api/orders | 주문 생성 | 201 `{orderId, orderNo, totalPrice, status:"결제대기"}` | 미로그인 401 / 배달조건·입력 400 |
| POST /api/orders/delivery-check | 배달 가능 여부 | 200 `{deliverable, reason}` | 미로그인 401 |
| GET /api/orders | 내 주문 목록 | 200 `[요약...]` | 미로그인 401 |
| GET /api/orders/{id} | 주문 상세(본인만) | 200 `{...items[]}` | 남의 주문 403 / 없음 404 |

### 보안 규칙 (핵심)
- **금액 서버 재계산**: 화면이 보낸 금액을 믿지 않고, 메뉴 DB에서 기본가+옵션가로 단가를 다시 계산해 합산. 주문 시점의 이름·단가를 항목에 저장(과거 주문 보존).
- **배달 조건 서버 재검증**: `deliveryAddress`가 "명지에코펠리스" 포함 + 샌드위치(category="샌드위치") 수량 합 ≥ 2. 미충족 시 거부.
- 로그인 필요(인증 필터의 회원 id 사용), 남의 주문 조회는 403.

### 설계 메모 (중요)
- 스펙은 "서버 장바구니(cart_item)에서 항목을 읽는다"지만, **서버 장바구니(05_API §3.3)가 아직 없어** 이번엔 주문 항목을 요청으로 받는다. 대신 금액은 서버가 DB로 재계산하므로 보안 원칙(조작 방지)은 지킨다. 서버 장바구니 도입 후 "장바구니에서 읽는" 방식으로 교체 예정. 쿠폰/적립은 이후 단계.

### 주요 파일 (server/.../order/)
- `OrderEntity`(orders)·`OrderItemEntity`(order_item)·`OrderRepository`
- `OrderService`(계산·배달검증·생성/목록/상세)·`OrderController`·`dto/*`

### 검증
- `gradlew build`(ASCII, H2) 성공. 새 테스트 `OrderControllerTest` 4건(미로그인401 / 생성201·총액25800 / 배달1개차단400 / 목록·상세) + 기존 전부 통과.
- MySQL 실측: 로그인 → 주문 생성(주문번호 `20260703-0001`, 총액 25,800 = 12,900×2) → 목록/상세 정상, 배달 1개 주문은 400 차단. `orders`/`order_item` 테이블 저장 확인.
